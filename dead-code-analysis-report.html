1. 误报分析:看似死代码实际有用的代码

以下文件在初步扫描中被标记为"死代码",但经深度验证确认实际正在被使用。这些是最容易被误删的文件,删除前必须二次确认。

1.1 admin/Menu.vue — 被 Layout.vue 相对路径导入

初步判定: 需确认 — grep 组件名"Menu"匹配过多,无法确定是否被引用

实际验证: 实际有用 — Layout.vue 第 37 行 import AdminMenu from './Menu.vue' 显式导入,第 7 行 <AdminMenu /> 使用。Layout.vue 被大量 admin 视图使用。

误报原因: "Menu" 是 Element Plus 图标名,grep 噪声大;相对路径 ./Menu.vue 未被全文组件名搜索覆盖。

1.2 四个 .styles.scss 文件 — 被 SCSS @use 语法引用

初步判定: 真死代码 — 搜索 import.*\.styles\.scss 零匹配

实际验证: 实际有用 — 以下 4 个文件通过 SCSS @use 语法被父 Vue 文件引入:

文件引用位置语法
Home.vue.styles.scssHome.vue:1383@use './Home.vue.styles';
User.vue.styles.scssUser.vue:549@use './User.vue.styles';
AgentDetail.vue.styles.scssAgentDetail.vue:974@use './AgentDetail.vue.styles';
Xuqiu.vue.styles.scssXuqiu.vue:312@use './Xuqiu.vue.styles';

误报原因: 初步扫描使用 JS import 语法搜索,未覆盖 SCSS 原生的 @use/@import 指令。SCSS 样式文件的引用关系必须用 SCSS 语法搜索。

注意: 另外 4 个 .styles.scss 文件(Vip.vue.styles.scss、Settings.vue.styles.scss、Orders.vue.styles.scss、Distribution.vue.styles.scss)经验证确实未被引用,父 Vue 文件仅 @use '@/styles/breakpoints',不引用自身 .styles.scss。这 4 个是真死代码。

1.3 del-nul.mjs — check-nul.mjs 的配套清理工具

初步判定: 需确认 — 不在 package.json scripts 中,无 CI 引用

实际验证: 实际有用 — check-nul.mjs(已接入 pre-commit hook)在检测到 nul 残留时,输出提示: "如需删除,请调用同目录的 del_nul.mjs"。del-nul.mjs 是用户手动执行的清理工具,非自动化调用。

误报原因: 工具间的"人肉调用"关系(控制台提示引导用户运行)不会被自动化 grep 覆盖。

1.4 STATE.md / loop-run-log.md / loop-budget.md / LOOP.md — CI 工作流活跃文件

初步判定: 真死代码 — 看起来是 loop 模式遗留的临时文件

实际验证: 实际有用 — 被 .github/workflows/loop-daily-triage.yml CI 工作流读写。该 workflow 注释明确: "只扫描+写 3 个 markdown: loop-run-log.md / STATE.md / loop-budget.md"。

误报原因: 文件名像临时产物,但实际是 CI 工作流的状态持久化文件。LOOP.md 虽当前目标已完成(COMPLETED),但仍被 CI 注释引用为"设计参考"。

1.5 根目录 nginx.conf — Dockerfile.client 生产构建引用

初步判定: 真死代码 — 与 client/nginx.conf 和 server/nginx.conf 重复

实际验证: 实际有用 — Dockerfile.client 第 29 行 COPY nginx.conf /etc/nginx/conf.d/default.conf 引用,用于生产 Docker 构建。三份 nginx.conf 用途不同:根=生产前端、client=本地开发、server=后端反代。

误报原因: 同名文件在多目录出现,容易被误判为重复。实际是不同环境的不同配置。

1.6 未在 __init__.py re-export 的 Python model 文件

初步判定: 需确认 — 不在 models/__init__.py 中导出

实际验证: 实际有用 — admin_models.py、certificate_models.py、order_models.py、vip_models.py 等 15+ 个 model 文件,均被对应域路由模块直接 import(如 app/api/v1/certificate/routes.py import certificate_models)。仅未在 __init__.py 中 re-export,不是死代码。

误报原因: __init__.py 的 re-export 是便利导出,不是模块存活的依据。Python 支持直接 import 任意模块路径。

2. 已建未接线:shadow/canary 影子流量子系统

这是一组功能完整但从未接入生产请求路径的代码,是本次分析中最值得关注的"灰色地带"。

子系统构成: 以下 5 个模块互相引用,形成完整的影子流量对比+金丝雀调比系统:

文件功能测试覆盖生产接线
shadow_traffic.pyShadowRouter: 影子流量路由器,按比例复制请求有无
shadow_compare.pyShadowCompareAggregator: 对比 v1/v2 响应差异有无
shadow_whitelist.py影子流量白名单过滤有无
shadow_ratio_controller.pyShadowRatioController: 基于错误率自动调比有无
canary_shadow_link.py金丝雀阶段与影子流量联动有无

验证结论:

  • main.py 中零处引用 shadow 模块
  • canary_routes.py(已注册路由)仅 import canary_stages 和 security,不引用任何 shadow 模块
  • canary_metrics.py 中的 sync_canary_shadow_all() 和 get_shadow_ratio_snapshot() 函数定义了但从未被调用
  • shadow 模块仅在 tests/test_shadow_*.py 和 tests/test_canary_shadow_link.py 中被引用
  • 模块间使用 TYPE_CHECKING 块和函数内 lazy import 互相引用,形成自包含集群

判定: 已建未接线 — 这是一个设计完整、实现完整、测试完整的影子流量系统,包含 Prometheus 指标(zhs_shadow_compare_total、zhs_shadow_ratio 等)、自动调比逻辑、白名单过滤。但从未被接入生产请求管道。

可能原因: 为 v1→v2 后端迁移准备的影子对比基础设施,迁移完成后未启用即搁置;或为未来金丝雀发布预留的能力。

建议: 不要直接删除。如果短期无启用计划,可标记为 # DEPRECATED: shadow traffic system, not wired into production 并移至 app/_archived/ 目录。如果确认永远不会启用,再删除。
