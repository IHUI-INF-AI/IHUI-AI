## 二、5 个核心微服务字段级对比

> 通用映射约定：
> - Java `Long` ↔ Python `BigInteger`（SQLAlchemy 大整数）
> - Java `String` ↔ Python `String(N)`
> - Java `Integer` ↔ Python `Integer`
> - Java `Boolean` ↔ Python `Boolean`
> - Java `LocalDateTime` ↔ Python `DateTime`
> - Java `BigDecimal` ↔ Python `Numeric`（learn/pay 用 `Integer(分)` 替代）
> - Java 枚举 ↔ Python `String`/`Integer`（按 comment 约定）

### 3. 通用风险点（贯穿所有服务）

1. **Long 主键 → String(64) UUID**：G 盘所有 `member_id`/`user_id` 改为 String(64) 存 UUID，而 H 盘 Long 存自增 ID。**这意味着数据迁移方案需要重新设计**，无法用主键直接对接。
2. **枚举 → String/Integer**：G 盘用 String(20) 或 Integer 表示原 Java 枚举，未保留枚举约束。
3. **BigDecimal → Integer(分)**：所有价格/金额字段在 G 盘统一改为 Integer 存分，Java 端 BigDecimal 元为单位的逻辑需改写。
4. **String → Text**：H 盘 String 字段在 G 盘被改为 Text（不限长度），需关注 MySQL 索引长度上限。
5. **Course 表不一致**：`course_models.py` 中 14 个 model 类实际来自 ZHS_Server_java 项目，与 ihui-ai-edu-learn-service 的 Lesson 无对应关系，**这是命名误导**。
6. **BaseEntity 公共字段**：所有 entity 都继承 BaseEntity 含 id/createTime/updateTime，G 盘统一使用 TimestampMixin 提供 created_at/updated_at（**字段名不一致**：createTime → created_at）。
7. **物理删除 vs 逻辑删除**：Java 端硬删除为主，G 盘部分表（Circle/CirclePost/AskQuestion/AskAnswer）增加了 `deleted` Boolean 字段做软删除。

按你提出的"封版上线、不新增功能"原则，本审计**仅做修复与一致性核对**，建议按以下顺序修复（**不是新增功能，仅补齐缺失字段**）：
