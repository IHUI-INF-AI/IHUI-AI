# IaC 决策:是否采用 K8s

> 决策日期:2026-07
> 决策状态:**已采纳** — 维持单机 Docker 部署
> 决策人:架构组
> 评审周期:每季度或触发条件命中时复审

---

## 1. 决策背景

### 1.1 旧架构 IaC 资产盘点

旧架构(重构前)共有 **51 个 IaC 文件**,分布如下:

| 类别 | 文件数 | 用途 |
|------|--------|------|
| Terraform | 5 | 基础设施资源声明(云资源、网络、数据库实例) |
| ArgoCD | 7 | GitOps 部署流水线与应用配置 |
| Helm | 20 | K8s 应用包管理与 values 配置 |
| K8s manifests | 19 | 原生 K8s 资源(deployment/service/ingress/configmap 等) |
| **合计** | **51** | — |

### 1.2 新架构当前部署形态

新架构(重构后)**不再使用 K8s/GitOps/Helm**,改为:

- **单机 Docker + docker-compose** 编排
- **Blue-Green 部署**通过 `deploy.sh` + `deploy_certs.sh` 脚本实现零停机切换
- 服务组件(共 6 个):`web` / `api` / `ai-service` / `postgres` / `redis` / `prometheus`

### 1.3 引发决策的问题

旧 IaC 资产已废弃,新架构是否需要**恢复 K8s + Helm + ArgoCD 的 GitOps 部署体系**?

---

## 2. 决策问题

> **是否在新架构中恢复 K8s/GitOps/Helm 部署体系?**

该决策直接影响:
- 运维复杂度与团队 skill 要求
- 部署频率与回滚机制
- 多环境(dev/staging/prod)管理方式
- 资源占用与成本结构

---

## 3. 选项分析

### 选项 A:维持单机 Docker(当前方案)

**优点**
- 部署链路简单,`docker-compose up -d` 即可启动
- 运维成本低,无需维护 etcd / control plane / Ingress Controller
- 启动快(秒级),资源占用低(无 K8s 控制面开销)
- 团队无需掌握 K8s/Helm/ArgoCD 体系
- 调试链路短,本地与生产环境一致性高

**缺点**
- 无自动扩缩容(HPA/VPA 不可用)
- 无自愈能力(容器挂掉需人工或脚本拉起)
- 单点故障风险(单机宕机即全站不可用)
- 蓝绿切换依赖 `sed` 修改 compose 文件,流程较脆弱
- 多环境一致性需手动维护

**适用场景**
- 中小规模(单机 QPS < 1000)
- MVP 验证阶段
- 研发/测试环境

### 选项 B:全量上 K8s + Helm + ArgoCD

**优点**
- 自动扩缩容(HPA 基于 CPU/内存/自定义指标)
- 自愈能力(容器异常自动重启/重调度)
- 原生滚动更新、金丝雀、蓝绿发布
- GitOps 实现声明式部署与审计追溯
- 多环境一致性(同一 chart 不同 values)

**缺点**
- 运维复杂度高,需 K8s 专家或托管集群(成本高)
- 学习曲线陡,Helm/ArgoCD/CRD 体系庞大
- 资源占用高(etcd + control plane 至少 2GB+ 内存)
- 本地开发链路变长(minikube/kind 仍有差异)
- IaC 重建工作量约 **3-5 人天**

**适用场景**
- 大规模(QPS > 5000)
- 多副本、多环境
- 生产高可用要求强

### 选项 C:混合方案(Docker 开发 + K8s 生产)

**优点**
- 开发环境保持简单(Docker Compose)
- 生产环境获得 K8s 高可用能力

**缺点**
- 两套配置维护成本高(docker-compose.yml + Helm chart)
- 环境差异风险:本地跑通 ≠ 生产跑通
- CI/CD 需维护双流水线
- 实际收益有限,复杂度却叠加

---

## 4. 决策结论

> **当前阶段(2026-07)维持选项 A:单机 Docker 部署。**

### 4.1 核心理由

1. **项目处于 MVP 验证阶段**,QPS 未达 K8s 迁移阈值(实测峰值远低于 1000 QPS)
2. **团队规模小**,引入 K8s 运维成本过高,会挤占业务研发资源
3. **当前 Docker Blue-Green 已满足零停机部署需求**,无迫切痛点驱动迁移
4. **IaC 重建工作量约 3-5 人天**,ROI 偏低,优先级低于核心业务功能
5. **单机资源占用可控**,K8s 控制面 2GB+ 内存开销对当前规模属浪费

### 4.2 决策边界

- 本决策**仅适用于 2026-07 至下一复审节点**
- 复审节点:季度评审 或 触发条件命中
- 一旦触发条件命中,立即启动 K8s 迁移评估,优先重建 Helm chart

## 5. 触发条件(重新评估迁移到 K8s)

> 满足以下**任一**条件时,立即启动 K8s 迁移评估流程。

| # | 触发条件 | 量化指标 | 数据来源 |
|---|----------|----------|----------|
| 1 | 单机 QPS 持续过高 | QPS > 1000,且单机 CPU > 70% 持续 1 周 | Prometheus `node_cpu_seconds_total` |
| 2 | 需多环境隔离 | 业务需要 dev/staging/prod 三套独立环境 | 产品需求 |
| 3 | 团队规模扩张 | 团队成员 > 10 人 | HR 数据 |
| 4 | 服务数量增长 | 微服务数 > 20 个(当前 6 个) | 架构盘点 |
| 5 | 高可用硬性要求 | 业务 SLA 要求 99.9%+,需多副本+自愈 | 商业合同/合规 |

**触发流程**
1. 触发条件命中 → 架构组发起 RFC
2. RFC 评估迁移工作量、成本、收益
3. 通过后启动 Helm chart 重建(web + api 优先)
