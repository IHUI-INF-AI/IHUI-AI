# 国际化(i18n)指南

> IHUI-AI 国际化体系总览:next-intl 5 语言、68 命名空间、12 守门/工作流脚本、品牌映射表的完整使用手册。翻译策略与守门规则边界见 [AGENTS.md §20](../AGENTS.md),补全计划历史见 [I18N-COMPLETION-PLAN.md](./I18N-COMPLETION-PLAN.md)。

---

## 1. 总览

IHUI-AI 前端 `apps/web` 采用 [next-intl](https://next-intl-docs.vercel.app/) 作为国际化框架,以 **`zh-CN` 简体中文为基准语言**,同步维护 `zh-TW`(繁体中文/台湾)、`ko`(韩语)、`ja`(日语)、`en`(英语)共 5 种语言。

- **翻译文件**:`apps/web/messages/<locale>.json`(5 个文件)
- **命名空间**:68 个顶层 namespace,嵌套 JSON 结构
- **请求配置**:`apps/web/src/i18n/request.ts`(基于 cookie 的 locale 解析)
- **守门规则**:详见 [AGENTS.md §20 i18n 约束规则](../AGENTS.md),本文档聚焦脚本工作原理与实操工作流,不重复规则条款

### 设计原则

1. **zh-CN 是唯一基准**:其他 4 语言必须与 zh-CN key 集合完全一致(parity)
2. **机械转换优先**:zh-TW 用 opencc-js 字形转换,ko/ja 用字符范围检测,减少人工
3. **品牌名 canonical 化**:中文品牌名在非中文 locale 统一替换为官方英文名
4. **守门前置**:pre-commit 钩子拦截简体残留 / 中文残留 / 破碎机翻,不让问题进仓库

## 2. 语言矩阵

| 语言 | 文件 | locale code | 检测策略 | 守门脚本 | 阻塞级别 |
|------|------|-------------|----------|----------|----------|
| 简体中文(基准) | `apps/web/messages/zh-CN.json` | `zh-CN` | — | — | 基准,不检测 |
| 繁体中文(台湾) | `apps/web/messages/zh-TW.json` | `zh-TW` | opencc-js 简→繁字形转换 | `scan-i18n-zh-residue.mjs zh-TW` | **阻塞**(exit 1) |
| 韩语 | `apps/web/messages/ko.json` | `ko` | 字符范围检测(汉字 + 无 Hangul) | `scan-i18n-zh-residue.mjs ko` | **阻塞**(exit 1) |
| 日语 | `apps/web/messages/ja.json` | `ja` | warnOnly(日文汉字词假阳性多) | `scan-i18n-zh-residue.mjs ja` | warn-only(exit 0) |
| 英语 | `apps/web/messages/en.json` | `en` | 破碎机翻检测 + 中文残留兜底 | `check-i18n-broken-en.mjs` | **阻塞**(exit 1) |

**全语言 parity 守门**:`check-i18n-keys.mjs` 以 zh-CN 为基准,检测其他 4 语言的 key 缺失/多余,`--staged` 模式 exit 1 阻塞 commit。

## 3. 命名空间组织

`zh-CN.json` 顶层有 68 个 namespace,按业务域分组。高频 namespace 易在多 agent 并发添加时产生重复键,新增前必须 Grep 确认。

### JSON 嵌套结构示例

```json
{
  "common": {
    "login": "登录",
    "orderStatus": {
      "pending": "待付款",
      "paid": "已付款",
      "shipped": "已发货"
    },
    "tools": {
      "title": "工具中心",
      "subtitle": "浏览并使用平台提供的各类工具"
    }
  },
  "models": {
    "nav": { "sort": "排序" },
    "market": { "title": "模型市场" }
  },
  "nav": {
    "dashboard": "仪表盘",
    "settings": "设置"
  }
}
```

### 高频命名空间(易冲突,新增 key 前必查)

| namespace | 用途 | 冲突风险 |
|-----------|------|----------|
| `nav` | 导航栏文案 | 高(多页面共享) |
| `models` | AI 模型市场 / 模型选择器 | 高(嵌套 nav/sort/market) |
| `sort` | 列表排序选项 | 高(多列表复用) |
| `market` | 市场 / 广场模块 | 高 |
| `common` | 通用按钮 / 状态文案 | 极高(几乎所有页面引用) |
| `settings` | 设置页 | 中 |
| `exam` / `learn` / `live` | 教育 / 学习 / 直播业务域 | 中 |

> ⚠️ **JSON 重复键禁止**:`JSON.parse` 遇到同级重复 key 静默取最后一个,前面被 shadowed。新增 key 前用 `Grep` 检查同级命名空间无同名块。`verify-i18n.mjs` 提供大小写敏感的重复键检测。

## 4. 翻译策略(实操)

品牌名 / 公司名 / 字体名 / 人名 / 技术术语的 canonical 映射规则见 [AGENTS.md §20 翻译策略](../AGENTS.md)。本节聚焦如何用脚本落地这些规则。

### 4.1 品牌 / 公司 / 字体 / 术语映射

`scripts/brand-glossary.json` 是机器可读的 canonical 映射表,合并 3 类映射:

| 类别 | 字段 | 示例 |
|------|------|------|
| 品牌 | `brands` | `智谱清言 → Zhipu AI`、`百度文心 → Baidu ERNIE`、`宇树科技 → Unitree` |
| 字体 | `fonts` | `宋体 → SimSun`、`黑体 → SimHei`、`微软雅黑 → Microsoft YaHei` |
| 术语 | `terms` | `物联网 → IoT`、`人工智能 → AI` |

**应用脚本**:`scripts/apply-brand-glossary.mjs`

```bash
# 1. 干跑预览将替换的 key + 旧值 + 新值(不写文件)
node scripts/apply-brand-glossary.mjs --dry-run

# 2. 实际写回 en/ko/ja 3 个语言文件
node scripts/apply-brand-glossary.mjs

# 3. 仅处理 en.json
node scripts/apply-brand-glossary.mjs --locale=en

# 4. zh-TW 需显式指定 + 人工核对(默认跳过,保留繁体品牌名)
node scripts/apply-brand-glossary.mjs --locale=zh-TW --dry-run
```

**安全限制**:
- 只处理 key 中含中文字符的映射条目,跳过纯英文 key(避免把 en.json 中合法的 `Moonshot` 错误替换为 `Moonshot AI`)
- value 是文件路径或 URL 时不替换(避免破坏 `/images/svg/大模型.svg`)
- 替换是字符串字面量替换(split/join),不解析 JSON

### 4.2 人名翻译

fictional / 示例数据人名用拼音:

| zh-CN | en | ko | ja |
|-------|----|----|-----|
| 李思涵 | Li Sihan | 리쓰한 | リ・スハン |
| 李总 | Manager Li | 이 대표 | 李社長 |

称呼符合目标语言习惯,不直译。

## 5. 守门工具详解

### 5.1 scan-i18n-zh-residue.mjs(通用中文残留守门)

替代旧的 `scan-zh-tw-simp.mjs` 与 `scan-ko-zh-residue.mjs` 两个专用脚本,可扩展到任意非中文 locale。

```bash
node scripts/scan-i18n-zh-residue.mjs <locale> [--staged] [--readme]
```

**按 locale 分支的检测逻辑**:

| locale | mode | 检测方法 | 失败条件 |
|--------|------|----------|----------|
| `zh-TW` | `opencc` | opencc-js `cn → tw` 字形转换,`converted !== value` 即简体残留 | 纯简体残留 → exit 1 |
| `ko` | `charRange` | value 含汉字 `[\u4e00-\u9fff]` 且不含 Hangul `[\uac00-\ud7af]` | 纯中文残留 → exit 1;半翻译(汉字+本地字符)→ warn |
| `ja` | `warnOnly` | 日文汉字词(登録/確認/削除等)假阳性海量,所有汉字只 warn | 始终 exit 0 |
| 其他 | `charRange`(`localRe=null`) | 任何含汉字即纯中文残留 | exit 1 |

**`LOCALE_CONFIG` 配置表**(脚本内):

```javascript
const LOCALE_CONFIG = {
  'zh-TW': { mode: 'opencc' },
  ko: { mode: 'charRange', localRe: /[\uac00-\ud7af]/ }, // 韩语 Hangul
  ja: { mode: 'warnOnly' },
}
```

**Markdown 模式 `--readme`**:扫描 `README.<locale>.md` 而非 `messages/<locale>.json`,跳过代码块 / HTML 注释 / 图片标签 / 链接 URL 部分,ICP 备案号行级豁免。**新增 locale 只需在 `LOCALE_CONFIG` 加一行**。

### 5.2 check-i18n-broken-en.mjs(破碎机翻英文守门)

检测 `en.json` 中的破碎机翻英文,基于历史修复案例归纳 4 类破碎模式:

| 检测类型 | 模式 | 示例 |
|----------|------|------|
| `no-space-concat` | CamelCase × 3+ 段无空格拼接 | `AgentDevPlatform`、`BigModelAppDev` |
| `case-chaos` | 单词内大小写异常切换 ≥3 次 | `M3SubAI`、`SubPayCTO` |
| `possible-pinyin` | 连续小写 ≥10 字符 + 非常见英文词 | `siningpeople`、`startpeople` |
| `zh-residue` | value 含中文字符(兜底) | 任何汉字残留 |

**豁免清单**(50+ 白名单 token):`AI`/`GPT`/`LLM`/`API`/`HTML`/`CSS`/`SDK`/`SaaS`/`OAuth`/`JWT`/`RAG`/`MCP`/`Copilot`/`Claude`/`Gemini` 等。纯大写单词(PLATFORM)、单词单大写开头、版本号/路径均豁免。

```bash
node scripts/check-i18n-broken-en.mjs              # 全量扫描
node scripts/check-i18n-broken-en.mjs --staged     # 仅 staged 改动(pre-commit)
node scripts/check-i18n-broken-en.mjs --fix        # 输出修复建议(不写文件)
node scripts/check-i18n-broken-en.mjs --readme     # 扫描 README.en.md
```

### 5.3 check-i18n-keys.mjs(key 完整性 + parity)

以 `zh-CN` 为基准,动态扫描 `apps/web/messages/*.json` 全部语言文件做 parity 检查。

```bash
node scripts/check-i18n-keys.mjs --staged   # pre-commit 用,有问题 exit 1
node scripts/check-i18n-keys.mjs            # 全量检查(CI 用),历史遗留 exit 0
```

**能力**:
- 扫描 `apps/web/` 下所有 `.ts/.tsx`(含 `app/`、`src/components/`、`src/lib/`),排除 `messages/`、`.next/`、`node_modules/`
- 同时识别 `useTranslations('ns')` 和 `getTranslations('ns')`(含 await)
- 单文件多命名空间:基于变量名精确归属,覆盖 `t`/`tc`/`te` 等变量;多 ns 时宽松检查(任一 ns 存在即通过)

### 5.4 brand-glossary.json(canonical 映射表)

机器可读的品牌 / 字体 / 术语映射表,供 `apply-brand-glossary.mjs` 与 review 流程引用。结构:

```json
{
  "version": "1.0.0",
  "updatedAt": "2026-07-19",
  "brands": { "智谱清言": "Zhipu AI", "...": "..." },
  "fonts":  { "宋体": "SimSun", "...": "..." },
  "terms":  { "物联网": "IoT", "...": "..." }
}
```

见 [§4.1](#41-品牌--公司--字体--术语映射)。

## 6. 翻译工作流脚本

除守门脚本外,`scripts/` 下还有 11 个 i18n 工作流脚本,覆盖"导出未翻译 → 批量翻译 → 应用 → 修复 → 校验"全链路。

| 脚本 | 用途 | 使用场景 |
|------|------|----------|
| `generate-i18n.js` | 基于旧架构翻译 + 简繁转换 + 翻译词典生成 ja/ko/zh-TW 三语言文件 | 初始化 / 大规模重建 |
| `translate-i18n-batch.mjs` | 用 StepFun AI 批量翻译未翻译键,收集唯一英文值调 API,写回翻译映射文件 | 大批量未翻译键处理 |
| `apply-i18n-translations.mjs` | 读取翻译映射(英文值 → 翻译值),将语言文件中值 === 英文原值的键替换为翻译值 | 配合 `translate-i18n-batch.mjs` 落地 |
| `sync-i18n-fixes.mjs` | 一键补全:把 zh-CN 新页面键复制到 ja/ko/zh-TW 补齐 parity + 补缺失键 | 新增页面后同步 |
| `export-untranslated-i18n.mjs` | 导出所有未翻译键(值 === en 仅 ASCII)到 `i18n-untranslated-<lang>.json` | 翻译前盘点 |
| `scan-zh-tw-untranslated.mjs` | 扫描 zh-TW.json 中漏译的英文 value,结果写入 `_scan_result.json` | zh-TW 专项漏译排查 |
| `fix-zh-tw-simp.mjs` | 用 opencc-js `cn → tw` 转换每个 value,自动写回 zh-TW.json | zh-TW 简体残留批量修复 |
| `fix-zhtw-parity.mjs` | 修复 zh-CN ↔ zh-TW parity 问题(hardcoded 命名空间中文 key 简繁不同) | parity 差异批量修复 |
| `fix-i18n-deep.mjs` | 配合 `deep-i18n-audit.mjs` 的问题报告,机械修复 zh-TW(字形+用词)/ ja(翻译表+模式)/ ko(翻译表+模式) | 深度审校问题修复 |
| `deep-i18n-audit.mjs` | 4 语言深度规则审校(zh-TW 繁简混淆 / ja 垃圾占位 / ko 乱码 / 跨语言一致性 / 术语统一) | 定期审校,有问题 exit 1 |
| `verify-i18n.mjs` | 验证 5 个 JSON 文件语法 + 检测重复键(替代 PowerShell `ConvertFrom-Json` 假阳性) | commit 前快速校验 |

### 典型翻译工作流

```bash
# 1. 盘点未翻译键
node scripts/export-untranslated-i18n.mjs

# 2. 批量翻译(需 StepFun API key)
STEPFUN_API_KEY=xxx node scripts/translate-i18n-batch.mjs

# 3. 应用翻译到语言文件
node scripts/apply-i18n-translations.mjs

# 4. 深度审校
node scripts/deep-i18n-audit.mjs --report

# 5. 机械修复审校问题
node scripts/fix-i18n-deep.mjs

# 6. 品牌 canonical 化
node scripts/apply-brand-glossary.mjs

# 7. 守门校验
node scripts/verify-i18n.mjs
node scripts/check-i18n-keys.mjs
node scripts/scan-i18n-zh-residue.mjs zh-TW
node scripts/scan-i18n-zh-residue.mjs ko
node scripts/check-i18n-broken-en.mjs
```

## 7. 添加新 key 流程

新增功能需要 i18n key 时,严格按以下流程:

```bash
# 1. 在 zh-CN.json 对应 namespace 加 key(基准语言)
#    先 Grep 确认同级无同名块,避免重复键

# 2. 跑 generate-i18n.js 同步到其他 4 语言(或手动复制 key,值留空/en 兜底)
node scripts/generate-i18n.js

# 3. 翻译未翻译键(见 §6 工作流)

# 4. 守门脚本验证
node scripts/check-i18n-keys.mjs --staged       # parity
node scripts/scan-i18n-zh-residue.mjs zh-TW --staged
node scripts/scan-i18n-zh-residue.mjs ko --staged
node scripts/check-i18n-broken-en.mjs --staged
```

> pre-commit 钩子会自动跑上述守门,本地验证通过后 commit 即可。

## 8. 添加新语言流程

新增 locale(如 `vi` 越南语)只需 3 步:

1. **在 `scan-i18n-zh-residue.mjs` 的 `LOCALE_CONFIG` 加一行**:

```javascript
const LOCALE_CONFIG = {
  'zh-TW': { mode: 'opencc' },
  ko: { mode: 'charRange', localRe: /[\uac00-\ud7af]/ },
  ja: { mode: 'warnOnly' },
  vi: { mode: 'charRange', localRe: /[\u00c0-\u024f]/ }, // 越南语拉丁扩展
}
```

2. **创建 `apps/web/messages/vi.json`**:从 `zh-CN.json` 复制结构,值翻译或留空

3. **在 `apps/web/src/i18n/request.ts` 的 `LOCALES` 数组加 `'vi'`**:

```typescript
const LOCALES = ['zh-CN', 'en', 'ja', 'ko', 'zh-TW', 'vi']
```

4. **更新相关脚本**:`translate-i18n-batch.mjs` / `apply-i18n-translations.mjs` / `export-untranslated-i18n.mjs` 的 `LANGS` 数组加 `'vi'`

## 9. README 多语言

根目录维护 4 语言 README,与 `apps/web/messages/` 的 locale 对齐:

| 文件 | 语言 | 守门脚本 |
|------|------|----------|
| `README.md` | 简体中文(基准) | — |
| `README.en.md` | 英语 | `check-i18n-broken-en.mjs --readme` + `scan-i18n-zh-residue.mjs en --readme` |
| `README.ko.md` | 韩语 | `scan-i18n-zh-residue.mjs ko --readme` |

**README 同步守门**:`scripts/check-readme-sync.mjs`(pre-commit 第 22 项,warn-only)检测 staged 中 `apps/*/src` 或 `packages/*/src` 功能代码改动但 `README.md` 未在 staged 中,提醒按 [AGENTS.md §22](../AGENTS.md) 同步更新 README。详见 [GATEKEEPERS.md](./GATEKEEPERS.md) 第 22 项。

## 10. 常见问题

### 10.1 JSON 重复键

**症状**:`JSON.parse` 不报错,但同级重复 key 静默取最后一个,前面被 shadowed,导致翻译"丢失"。

**排查**:
```bash
node scripts/verify-i18n.mjs   # 检测大小写敏感的重复键
```

**修复**:手动合并重复 key,删除多余的同名块。新增 key 前用 `Grep` 检查同级命名空间。

### 10.2 zh-TW 简体字残留

**症状**:zh-TW.json 中出现简体字(如 `智能` 而非 `智能`),opencc 字形转换检测到 `converted !== value`。

**修复**:
```bash
# 批量机械修复(字形转换,不改用词)
node scripts/fix-zh-tw-simp.mjs

# 深度修复(字形 + 27 条 TW 用词映射)
node scripts/fix-i18n-deep.mjs
```

### 10.3 ko 中文残留

**症状**:ko.json 中 value 含汉字且不含 Hangul,字符范围检测拦截。

**修复**:用 `translate-i18n-batch.mjs` + `apply-i18n-translations.mjs` 批量翻译,或手动替换。

### 10.4 en 破碎机翻

**症状**:en.json 出现 `AgentDevPlatform` / `M3SubAI` / `siningpeople` 等机翻拼接。

**修复**:
```bash
node scripts/check-i18n-broken-en.mjs --fix   # 输出修复建议
# 手动按建议修改 en.json
```

### 10.5 命名空间冲突

**症状**:多 agent 并发添加同名 namespace(如各自加了 `models.nav.sort`),JSON.parse 静默 shadow。

**预防**:
1. 新增 key 前 `Grep` 确认同级无同名块
2. 高频 namespace(nav/models/sort/market/common)优先复用,不新建
3. `verify-i18n.mjs` 检测大小写敏感重复键

### 10.6 zh-CN ↔ zh-TW parity 差异

**症状**:`hardcoded` 命名空间用中文文本作 key,简繁版本 key 不同导致 parity 差异(曾出现 1134 个差异)。

**修复**:
```bash
node scripts/fix-zhtw-parity.mjs   # 用 zh-CN 的 hardcoded 覆盖 zh-TW
```

> simplified 对 TW 用户可理解,优先保证 parity。

## 11. 前端使用

### 11.1 useTranslations hook

```typescript
'use client'
import { useTranslations } from 'next-intl'

export function LoginButton() {
  const t = useTranslations('common')
  return <button>{t('login')}</button>
}
```

### 11.2 getTranslations(服务端组件)

```typescript
import { getTranslations } from 'next-intl/server'

export default async function Page() {
  const t = await getTranslations('nav')
  return <h1>{t('dashboard')}</h1>
}
```

### 11.3 request.ts 配置

`apps/web/src/i18n/request.ts` 基于 cookie 解析 locale:

```typescript
import { getRequestConfig } from 'next-intl/server'
import { cookies } from 'next/headers'

const LOCALES = ['zh-CN', 'en', 'ja', 'ko', 'zh-TW']

export default getRequestConfig(async () => {
  const cookieStore = await cookies()
  const cookie = cookieStore.get('locale')?.value
  const locale = cookie && LOCALES.includes(cookie) ? cookie : 'zh-CN'
  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default,
  }
})
```

**locale 切换**:前端通过设置 `locale` cookie 切换语言,`LOCALES` 白名单外的值回退到 `zh-CN`。

### 11.4 middleware.ts

`apps/web/middleware.ts` 主要负责 SSO 鉴权与分域路由,**不做 locale 路由**(locale 走 cookie 而非 URL 路径)。locale 切换由客户端写 cookie + 刷新实现。

> next-intl 通过 `createNextIntlPlugin('./src/i18n/request.ts')` 在 `next.config.ts` 注册,见 `apps/web/next.config.ts` 末尾。
