# i18n 补全计划 (M-31)

## 1. 当前状态

### 1.1 新项目 (IHUI-AI)

| 指标 | 数值 |
|------|------|
| 语言文件 | `zh-CN.json` |
| 键数量 | 3,606 |
| 覆盖范围 | 核心功能（认证、课程、考试、证书、支付、管理后台等） |
| 支持语言 | zh-CN（简体中文） |

### 1.2 旧项目

| 指标 | 数值 |
|------|------|
| 键数量 | 11,888 |
| 支持语言 | zh-CN、en-US 等 |

### 1.3 差距分析

| 指标 | 数值 |
|------|------|
| 缺失键数 | 8,282 (11,888 - 3,606) |
| 覆盖率 | 约 30.3% |

## 2. 补全策略

### 2.1 当前阶段：仅服务中国市场

当前 3,606 个键已覆盖以下核心功能：

- 用户认证与身份验证（auth、auth-identity）
- 课程学习与报名（learn、edu-public）
- 考试系统（exam）
- 证书发放（certificate）
- 支付与订单（payment、order、billing）
- 分块上传（chunked-upload）
- 管理后台（admin、audit）
- 社区与互动（community、comments、plaza）

**结论**：当前阶段 3,606 键足以支撑国内市场的核心业务流程，暂不需要补充缺失的 8,282 个键。

### 2.2 出海阶段：需要补充的步骤

当项目计划扩展到海外市场时，按以下步骤执行：

#### 步骤 1：键值盘点与分类

- 对比旧项目 `zh-CN.json`（11,888 键）与新项目 `zh-CN.json`（3,606 键）
- 将缺失的 8,282 个键按模块分类：
  - **必须补充**：用户可见的 UI 文本、错误提示、邮件模板
  - **建议补充**：帮助文档、FAQ、政策页面
  - **可忽略**：已废弃功能的文案、旧版兼容文本

#### 步骤 2：zh-CN 补全

- 从旧项目迁移仍需要的中文键值
- 清理废弃键（合并重复、移除过期内容）
- 目标：zh-CN 键数量达到 8,000-10,000（根据实际功能需求）

#### 步骤 3：目标语言翻译

- 确定目标语言（建议优先：en-US、ja-JP、ko-KR）
- 使用 AI 辅助翻译工具批量生成初版翻译
- 人工校对关键内容（法律文本、支付条款、隐私政策等）

#### 步骤 4：集成与测试

- 将翻译文件集成到 `next-intl` 体系
- 运行 i18n 完整性检查，确保所有语言文件键值一致
- 在测试环境中切换语言验证显示效果

#### 步骤 5：持续维护

- 建立翻译工作流：新增键 → 自动通知 → 翻译 → 合并
- 定期审计 i18n 文件，清理未使用的键

### 2.3 AI 辅助翻译工具建议

| 工具 | 适用场景 | 说明 |
|------|----------|------|
| DeepL API | 高质量翻译 | 适合欧洲语言，API 可批量调用 |
| Google Cloud Translation | 多语言批量翻译 | 支持 100+ 语言，适合大规模迁移 |
| OpenAI GPT-4 / Claude | 上下文感知翻译 | 适合需要理解上下文的 UI 文案 |
| i18n-ally (VS Code 插件) | 开发阶段辅助 | 实时预览、缺失键检测、内联翻译 |

**推荐方案**：

1. 使用 AI 大模型（GPT-4 / Claude）生成初版翻译，因为 UI 文案需要理解上下文
2. 使用 i18n-ally 插件进行开发阶段的持续检查
3. 关键内容（法律、支付）由专业译员人工校对

## 3. 执行优先级

| 优先级 | 任务 | 触发条件 |
|--------|------|----------|
| P0 | 维护现有 3,606 键的完整性 | 持续进行 |
| P1 | 补全 zh-CN 缺失的核心键 | 新功能开发时同步补充 |
| P2 | 生成 en-US 翻译文件 | 确定出海时间表后启动 |
| P3 | 补全全部缺失键并支持多语言 | 海外市场正式运营前 |

## 4. 备注

- 本文档为 M-31 迁移项的补全计划，当前不需要立即执行翻译工作
- 新项目使用 `next-intl` 作为 i18n 框架，翻译文件位于 `apps/web/src/i18n/messages/`
- 旧项目数据仅作为参考，实际键值需根据新项目功能需求裁剪
